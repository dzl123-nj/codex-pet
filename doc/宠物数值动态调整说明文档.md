# 宠物数值动态调整说明文档

## 1. 概述

本系统是一个虚拟宠物养成系统，通过模拟真实宠物的生理和心理需求，实现宠物数值的动态调整。宠物拥有饥饿值、能量、清洁度、幸福感、健康值五大核心属性，这些属性会随时间自然衰减，并通过玩家的交互操作（喂食、清洁、睡眠、玩耍等）得到恢复或消耗。系统通过多种算法确保宠物数值在合理范围内波动，并根据属性值触发状态转换（如健康、生病、死亡等），同时提供经验值和等级成长机制。

### 1.1 系统架构概览

数值调整的核心逻辑集中在 `PetAttributeServiceImpl` 服务类中，通过以下六大算法协同运作：

1. **时间衰减算法** — `applyTimeDecay()`：模拟宠物随时间流逝的自然消耗
2. **行为交互算法** — `feed()` / `clean()` / `sleep()` / `play()` / `chatReward()` / `useItem()`：响应玩家操作
3. **状态转换算法** — `recalculateStatus()`：基于属性值判定宠物生死状态
4. **经验升级算法** — `checkLevelUp()` / `calculateExpForLevel()`：管理宠物成长
5. **数值边界算法** — `clamp()`：确保所有属性在合法范围内
6. **变更记录算法** — `recordChange()`：记录每次属性变动的审计日志

## 2. 核心属性体系

宠物拥有以下核心属性，所有属性值范围均为 **0 ~ 100**：

| 属性 | 字段名 | 说明 | 默认值 |
|------|--------|------|--------|
| **饥饿值** | hunger | 宠物的饱腹程度，过低会生病甚至死亡 | 50 |
| **能量** | energy | 宠物的精力水平，低于阈值会加速健康恶化 | 50 |
| **清洁度** | cleanliness | 宠物的卫生状况，低于阈值会触发连锁衰减 | 50 |
| **幸福感** | happiness | 宠物的情绪状态，受多因素综合影响 | 50 |
| **健康值** | health | 宠物的健康状况，降至0则宠物死亡 | 50 |
| **经验值** | experience | 累积经验值，用于等级提升 | 0 |
| **等级** | level | 宠物等级，范围 1 ~ 100 | 1 |

### 2.1 属性之间的关联关系

五大属性并非独立运作，它们之间存在复杂的关联关系：

- **饥饿值过低** → 幸福感加速衰减 + 健康值加速衰减 + 可能触发生病
- **清洁度过低** → 幸福感加速衰减 + 健康值加速衰减 + 可能触发生病
- **能量过低** → 幸福感加速衰减 + 健康值加速衰减 + 可能触发生病 + 无法玩耍
- **健康值过低** → 生病概率大幅上升 → 健康值进一步加速衰减（恶性循环）
- **健康值归零** → 宠物死亡，所有交互停止

这种设计模拟了真实宠物的"忽视照护 → 恶性循环"机制，鼓励玩家持续关注宠物的各项需求。

## 3. 算法一：时间衰减算法（applyTimeDecay）

### 3.1 算法概述

时间衰减算法是整个数值系统的基础驱动力。每当玩家访问宠物页面时，系统会根据宠物上次更新时间与当前时间的差值，批量计算所有属性的自然衰减量。这是一种**惰性计算（Lazy Evaluation）**模式——不在后台定时任务中逐秒计算，而是在玩家请求时一次性计算累积衰减。

### 3.2 算法流程

```
输入：VirtualPet pet（宠物对象）
输出：boolean（是否有属性发生变化）

步骤1：前置校验
  - 若 pet.status == null 或 pet.status == 3（死亡），直接返回 false，不做任何计算

步骤2：计算经过时间
  - now = 当前服务器时间戳（毫秒）
  - lastUpdate = 宠物上次更新时间（毫秒），若为null则取now
  - elapsedMs = now - lastUpdate
  - elapsedHours = elapsedMs / (1000 × 60 × 60)

步骤3：时间窗口校验
  - 若 elapsedMs < 60,000（即小于1分钟），返回 false，避免频繁微调
  - 若 elapsedHours > 72，截断为72小时，防止长时间离线导致属性过度衰减

步骤4：判断睡眠状态
  - isSleeping = (pet.status == 0)

步骤5：计算各项属性衰减/恢复量并更新
步骤6：记录变更日志
步骤7：重新计算宠物状态
步骤8：检查睡眠自动唤醒
```

### 3.3 各属性衰减公式详解

#### 3.3.1 饥饿值衰减

饥饿值是所有状态下都会持续衰减的属性，不受睡眠状态影响：

```
衰减量 = round(HUNGER_DECAY_PER_HOUR × elapsedHours) = round(2 × elapsedHours)
新饥饿值 = clamp(旧饥饿值 - 衰减量)
```

**示例**：玩家离开6小时后返回
- 衰减量 = round(2 × 6) = 12
- 若旧饥饿值为80，新饥饿值 = clamp(80 - 12) = 68
- 若旧饥饿值为5，新饥饿值 = clamp(5 - 12) = 0（被边界截断）

#### 3.3.2 能量衰减与恢复

能量的计算分为两种情况——清醒状态和睡眠状态：

**清醒状态（isSleeping = false）**：
```
能量衰减量 = round(ENERGY_DECAY_PER_HOUR × elapsedHours) = round(3 × elapsedHours)
能量恢复量 = 0
新能量 = clamp(旧能量 - 能量衰减量)
```

**睡眠状态（isSleeping = true）**：
```
能量衰减量 = 0（睡眠期间不衰减）
能量恢复量 = round((ENERGY_DECAY_PER_HOUR + 2) × elapsedHours) = round(5 × elapsedHours)
新能量 = clamp(旧能量 - 0 + 能量恢复量) = clamp(旧能量 + 能量恢复量)
```

**设计意图**：睡眠状态的恢复速率为5点/小时（衰减系数3 + 额外系数2），这意味着睡满10小时可恢复50点能量。

**示例**：宠物能量为20时睡眠8小时
- 能量恢复量 = round(5 × 8) = 40
- 新能量 = clamp(20 + 40) = 60

#### 3.3.3 清洁度衰减

清洁度在所有状态下都会衰减：

```
衰减量 = round(CLEANLINESS_DECAY_PER_HOUR × elapsedHours) = round(2 × elapsedHours)
新清洁度 = clamp(旧清洁度 - 衰减量)
```

#### 3.3.4 幸福感衰减（连锁计算）

幸福感的衰减是一个**依赖型计算**——它取决于饥饿值、清洁度和能量**衰减后的新值**。计算顺序非常重要：必须先计算完饥饿值、清洁度和能量的衰减，再根据它们的新值计算幸福感衰减。

```
基础衰减量 = HAPPINESS_DECAY_PER_HOUR × elapsedHours = 1 × elapsedHours

// 连锁惩罚：基于其他属性衰减后的新值
if 新饥饿值 ≤ 15:
    基础衰减量 += 1 × elapsedHours    // 饥饿惩罚

if 新清洁度 ≤ 15:
    基础衰减量 += 1 × elapsedHours    // 肮脏惩罚

if 新能量 ≤ 10 且 非睡眠状态:
    基础衰减量 += 1 × elapsedHours    // 疲惫惩罚

新幸福感 = clamp(旧幸福感 - round(基础衰减量))
```

**关键点**：连锁惩罚是叠加的。如果宠物同时饥饿、肮脏且疲惫，幸福感的衰减速率将变为基础值的4倍（1 + 1 + 1 + 1 = 4点/小时）。

**示例**：宠物饥饿值=10，清洁度=8，能量=5（清醒），离开4小时
- 基础衰减量 = 1 × 4 = 4
- 饥饿惩罚：10 ≤ 15 → +4
- 肮脏惩罚：8 ≤ 15 → +4
- 疲惫惩罚：5 ≤ 10 且清醒 → +4
- 总衰减量 = round(4 + 4 + 4 + 4) = 16
- 若旧幸福感为50，新幸福感 = clamp(50 - 16) = 34

#### 3.3.5 健康值衰减（连锁计算）

健康值的衰减是最复杂的，它同时受到睡眠状态、其他属性阈值和生病状态的影响：

```
// 基础衰减：睡眠时不衰减
if isSleeping:
    基础衰减量 = 0
else:
    基础衰减量 = HEALTH_DECAY_PER_HOUR × elapsedHours = 1 × elapsedHours

// 连锁惩罚：基于其他属性衰减后的新值
if 新饥饿值 ≤ 15:
    基础衰减量 += 3 × elapsedHours    // 饥饿对健康影响最大（系数3）

if 新清洁度 ≤ 15:
    基础衰减量 += 2 × elapsedHours    // 肮脏对健康影响次之（系数2）

if 新能量 ≤ 10 且 非睡眠状态:
    基础衰减量 += 1 × elapsedHours    // 疲惫对健康影响较轻（系数1）

if 当前状态 == 2（生病）:
    基础衰减量 += 3 × elapsedHours    // 生病本身加速健康恶化

// 睡眠恢复
if isSleeping:
    健康恢复量 = round(HEALTH_DECAY_PER_HOUR × elapsedHours) = round(1 × elapsedHours)
else:
    健康恢复量 = 0

新健康值 = clamp(旧健康值 - round(基础衰减量) + 健康恢复量)
```

**设计意图**：
- 饥饿对健康的伤害最大（系数3），体现"饥饿致命"的设计理念
- 清洁对健康的伤害次之（系数2），体现"卫生影响健康"
- 疲惫对健康的伤害最轻（系数1），体现"疲劳会慢慢拖垮身体"
- 生病状态下健康加速恶化（系数3），模拟"久病不愈的恶性循环"
- 睡眠时健康缓慢恢复（系数1），体现"休息是最好的疗养"

**最坏情况示例**：宠物饥饿值=10，清洁度=8，能量=5，生病状态，离开10小时
- 基础衰减量 = 1×10 + 3×10 + 2×10 + 1×10 + 3×10 = 10 + 30 + 20 + 10 + 30 = 100
- 健康恢复量 = 0（清醒状态）
- 若旧健康值为60，新健康值 = clamp(60 - 100) = 0 → **宠物死亡！**

### 3.4 睡眠自动唤醒机制

在时间衰减计算完成后，系统会检查宠物是否应该自动唤醒：

```
if isSleeping 且 新能量 ≥ 100:
    pet.status = 1（清醒）
```

这意味着宠物会在能量恢复到满值时自动醒来，无需玩家手动唤醒。

### 3.5 变更记录

所有属性变化都会被记录到 `pet_attribute_record` 表中，包括：
- 属性类型（hunger / energy / cleanliness / happiness / health）
- 变化前后的具体数值
- 变化原因（"自然衰减" / "睡眠恢复" / "生病衰减"）

## 4. 算法二：行为交互算法

### 4.1 喂食算法（feed）

#### 4.1.1 前置校验流程

```
校验1：pet.status == 3（死亡）→ 拒绝，返回"宠物已经不在了"
校验2：pet.status == 0（睡眠）→ 拒绝，返回"宠物正在睡觉"
校验3：旧饥饿值 > 90 → 拒绝，返回"已经吃得很饱了"（防过度喂食）
```

#### 4.1.2 数值调整公式

```
新饥饿值 = clamp(旧饥饿值 + 30)
新幸福感 = clamp(旧幸福感 + 15)
```

#### 4.1.3 后续处理

```
recalculateStatus(pet)    // 重新判定宠物状态
savePet(pet)              // 持久化到数据库
```

#### 4.1.4 反馈消息算法

系统根据喂食后的饥饿值返回不同语气的反馈：

```
if 新饥饿值 ≥ 80:
    "谢谢你给我喂食！我吃得好饱好满足呀~"
else if 新饥饿值 ≥ 50:
    "谢谢你给我喂食！嗯~ 好好吃！"
else:
    "谢谢你给我喂食！还要再吃一点...还是有点饿"
```

### 4.2 清洁算法（clean）

#### 4.2.1 前置校验

```
校验1：pet.status == 3（死亡）→ 拒绝
校验2：pet.status == 0（睡眠）→ 拒绝
```

#### 4.2.2 数值调整公式

```
新清洁度 = clamp(旧清洁度 + 40)
新健康值 = clamp(旧健康值 + 5)
新幸福感 = clamp(旧幸福感 + 5)
```

**设计意图**：清洁操作不仅恢复清洁度，还能小幅恢复健康值和幸福感，体现"清洁卫生有益身心健康"。

#### 4.2.3 反馈消息算法

```
if 新清洁度 ≥ 80:
    "洗完澡浑身香喷喷的！你看我多干净~"
else:
    "谢谢你帮我洗澡，舒服多啦~"
```

### 4.3 睡眠算法（sleep）

#### 4.3.1 前置校验

```
校验1：pet.status == 3（死亡）→ 拒绝
校验2：pet.status == 0（已在睡眠）→ 拒绝
```

#### 4.3.2 数值调整公式

```
新能量 = clamp(旧能量 + 50)
新幸福感 = clamp(旧幸福感 + 5)
新经验值 = 旧经验值 + 5（经验值不受clamp限制）
pet.status = 0（进入睡眠状态）
```

#### 4.3.3 后续处理

```
checkLevelUp(pet)    // 检查是否升级
savePet(pet)         // 持久化
```

#### 4.3.4 反馈消息算法

```
if 新能量 ≥ 90:
    "精力充沛！睡了一觉感觉好极了~"
else if 新能量 ≥ 60:
    "嗯...打了个盹，感觉好多了"
else:
    "好困...让我再睡一会儿...Zzz"
```

### 4.4 唤醒算法（wake）

#### 4.4.1 前置校验

```
校验1：pet.status == 3（死亡）→ 拒绝
校验2：pet.status != 0（未在睡眠）→ 拒绝
```

#### 4.4.2 状态变更

```
pet.status = 1（清醒）
recalculateStatus(pet)    // 唤醒后立即判定是否生病
savePet(pet)
```

#### 4.4.3 反馈消息算法

```
if 能量 ≥ 70:
    "伸个懒腰~ 我醒啦！今天精力充沛！"
else:
    "嗯...被叫醒了...还想再睡一会儿..."
```

### 4.5 玩耍算法（play）

#### 4.5.1 前置校验

```
校验1：pet.status == 3（死亡）→ 拒绝
校验2：pet.status == 0（睡眠）→ 拒绝
校验3：当前能量 < 15 → 拒绝，返回"宠物太累了"（能量不足以玩耍）
```

#### 4.5.2 数值调整公式

玩耍是所有交互中**影响属性最多**的操作——它同时影响4个属性：

```
新幸福感 = clamp(旧幸福感 + 25)    // 玩耍带来快乐
新能量   = clamp(旧能量 - 15)      // 玩耍消耗精力
新饥饿值 = clamp(旧饥饿值 - 5)     // 玩耍消耗体力，容易饿
新清洁度 = clamp(旧清洁度 - 5)     // 玩耍会弄脏
新经验值 = 旧经验值 + customExp     // 默认15，可自定义
```

**设计意图**：玩耍是"高收益高风险"的操作——获得大量幸福感和经验值，但同时消耗能量、饥饿值和清洁度。玩家需要在玩耍和休息/喂食之间做出策略性决策。

#### 4.5.3 支持自定义经验值

玩耍方法提供了重载版本 `play(pet, customExp)`，允许外部调用时传入自定义经验值。当 `customExp > 0` 时使用自定义值，否则使用默认值15。

#### 4.5.4 后续处理

```
checkLevelUp(pet)          // 检查是否升级
recalculateStatus(pet)     // 重新判定状态（玩耍后可能触发生病）
savePet(pet)
```

#### 4.5.5 反馈消息算法

```
if 新幸福感 ≥ 80:
    "和你一起玩好开心呀！我最喜欢你了！"
else:
    "嘿嘿，玩耍真有趣！还要玩还要玩~"
```

### 4.6 聊天奖励算法（chatReward）

#### 4.6.1 数值调整公式

```
新经验值   = 旧经验值 + 8
新幸福感   = clamp(旧幸福感 + 5)

// 能量消耗仅在清醒状态生效
if pet.status ≠ 0 且 pet.status ≠ 3:
    新能量 = clamp(旧能量 - 2)
```

**设计意图**：聊天是一种轻量级互动，消耗少、收益低，适合在两次主要操作之间维持宠物状态。睡眠和死亡状态下不消耗能量。

### 4.7 使用道具算法（useItem）

#### 4.7.1 前置校验

```
校验1：pet.status == 3（死亡）→ 拒绝
校验2：pet.status == 0（睡眠）→ 拒绝
```

#### 4.7.2 动态属性调整公式

道具的属性调整是**动态的**——每个道具可独立配置对各项属性的影响值：

```
if item.hungerChange != null:
    新饥饿值 = clamp(旧饥饿值 + item.hungerChange)

if item.happinessChange != null:
    新幸福感 = clamp(旧幸福感 + item.happinessChange)

if item.energyChange != null:
    新能量 = clamp(旧能量 + item.energyChange)

if item.cleanlinessChange != null:
    新清洁度 = clamp(旧清洁度 + item.cleanlinessChange)

if item.healthChange != null:
    新健康值 = clamp(旧健康值 + item.healthChange)

if item.experienceReward != null:
    新经验值 = 旧经验值 + item.experienceReward
```

**设计意图**：道具系统提供了极大的灵活性，可以设计出各种功能的道具——回复药水（+健康值）、超级食物（+饥饿值+幸福感）、能量饮料（+能量）等。

#### 4.7.3 后续处理

```
checkLevelUp(pet)
recalculateStatus(pet)
savePet(pet)
```

## 5. 算法三：状态转换算法（recalculateStatus）

### 5.1 宠物状态定义

| 状态码 | 状态名称 | 说明 |
|--------|----------|------|
| 0 | 睡眠 | 宠物正在睡觉，能量和健康恢复中 |
| 1 | 清醒 | 宠物正常活动状态，可进行各种交互 |
| 2 | 生病 | 宠物生病，健康值加速衰减，属性衰减加剧 |
| 3 | 死亡 | 宠物健康值归零，所有交互停止，不可恢复 |

### 5.2 状态转换算法流程

状态转换算法是一个**优先级判定链**，按优先级从高到低依次检查：

```
输入：VirtualPet pet
输出：更新 pet.status

步骤1：死亡状态不可逆
  if pet.status == 3:
      return    // 已死亡，不再做任何判定

步骤2：死亡判定（最高优先级）
  if pet.health ≤ 0:
      pet.status = 3
      pet.health = 0
      return    // 死亡后立即返回

步骤3：睡眠状态保护
  if pet.status == 0:
      return    // 睡眠期间不触发生病判定

步骤4：硬性生病触发（饥饿值阈值）
  if pet.hunger ≤ 15:
      pet.status = 2
      return    // 过度饥饿直接生病

步骤5：硬性生病触发（清洁度阈值）
  if pet.cleanliness ≤ 15:
      pet.status = 2
      return    // 过度肮脏直接生病

步骤6：概率性生病判定（综合评估）
  // 进入复杂的概率计算流程...
```

### 5.3 概性生病判定算法详解

当宠物不满足硬性生病条件时，系统通过概率模型判定是否生病。这是一个**多因子加权概率模型**。

#### 5.3.1 基础生病概率计算

基础概率仅与健康值相关，采用**分段线性函数**：

```
if health < 40:
    // 极低健康区间：概率从0.70起步，健康值每降1点概率+1%
    sickChance = 0.70 + (40 - health) × 0.01
    // health=0时: sickChance = 0.70 + 40×0.01 = 1.10
    // health=10时: sickChance = 0.70 + 30×0.01 = 1.00
    // health=20时: sickChance = 0.70 + 20×0.01 = 0.90
    // health=30时: sickChance = 0.70 + 10×0.01 = 0.80
    // health=39时: sickChance = 0.70 + 1×0.01 = 0.71

else if health < 60:
    // 中低健康区间：概率从0.15起步，健康值每降1点概率+1.25%
    sickChance = 0.15 + (60 - health) × 0.0125
    // health=40时: sickChance = 0.15 + 20×0.0125 = 0.40
    // health=50时: sickChance = 0.15 + 10×0.0125 = 0.275
    // health=59时: sickChance = 0.15 + 1×0.0125 = 0.1625

else if health < 80:
    // 中高健康区间：健康值每降1点概率+0.5%
    sickChance = (80 - health) × 0.005
    // health=60时: sickChance = 20×0.005 = 0.10
    // health=70时: sickChance = 10×0.005 = 0.05
    // health=79时: sickChance = 1×0.005 = 0.005

else:
    // 高健康区间（≥80）：不会生病
    sickChance = 0
```

**概率分布表**（仅健康值影响）：

| 健康值 | 基础生病概率 | 风险等级 |
|--------|-------------|----------|
| 0 ~ 10 | 100%+（截断为100%） | 必定生病 |
| 11 ~ 20 | 90% ~ 99% | 极高风险 |
| 21 ~ 30 | 80% ~ 89% | 高风险 |
| 31 ~ 39 | 71% ~ 79% | 较高风险 |
| 40 ~ 49 | 27.5% ~ 40% | 中等风险 |
| 50 ~ 59 | 16.25% ~ 27.5% | 较低风险 |
| 60 ~ 69 | 5.5% ~ 10% | 低风险 |
| 70 ~ 79 | 0.5% ~ 5% | 极低风险 |
| 80 ~ 100 | 0% | 安全 |

#### 5.3.2 额外风险因子加成

在基础概率之上，系统叠加三个额外风险因子：

```
if hunger < 30:
    sickChance += 0.10    // 饥饿风险：+10%

if cleanliness < 30:
    sickChance += 0.10    // 肮脏风险：+10%

if energy < 15:
    sickChance += 0.10    // 疲惫风险：+10%

// 最终概率上限为100%
sickChance = min(1.0, sickChance)
```

**设计意图**：即使健康值较高（如75，基础概率仅2.5%），如果同时饥饿、肮脏且疲惫，总概率仍可达2.5% + 10% + 10% + 10% = 32.5%，体现"多重不利因素叠加"的风险放大效应。

#### 5.3.3 伪随机数生成与判定

系统使用**确定性伪随机数**（而非真随机）来决定是否生/恢复，确保公平性和可重现性：

```
// 时间种子：以30分钟为一个时间窗口
timeSeed = currentTimeMillis / (30 × 60 × 1000)

// 构造种子：宠物ID × 31 + 时间段
random = new Random(pet.id × 31 + timeSeed)
roll = random.nextDouble()    // 生成 [0, 1) 之间的随机数
```

**种子设计的精妙之处**：
- `pet.id × 31`：不同宠物在同一时间窗口内有不同的随机序列（31是质数，减少碰撞）
- `timeSeed`：每30分钟更新一次，同一宠物在30分钟内的多次请求得到相同结果
- 这意味着玩家无法通过频繁刷新页面来"刷"状态变更

#### 5.3.4 生病/恢复判定逻辑

```
if 已经生病（status == 2）:
    // 生病状态：尝试恢复
    // 恢复条件更严格：随机数必须 ≥ sickChance × 1.5
    if roll ≥ sickChance × 1.5:
        pet.status = 1（恢复健康）
    // 否则继续生病

else:
    // 健康状态：检查是否生病
    if roll < sickChance:
        pet.status = 2（生病）
    // 否则保持健康
```

**恢复概率分析**：

生病状态下恢复的概率 = `1 - sickChance × 1.5`（当 `sickChance × 1.5 ≤ 1` 时）

| 生病概率 | 恢复概率 | 说明 |
|----------|----------|------|
| 0%（健康值≥80） | 100% | 必定恢复 |
| 10% | 85% | 容易恢复 |
| 30% | 55% | 约一半概率恢复 |
| 50% | 25% | 不易恢复 |
| 66.7% | 0% | 几乎无法恢复 |
| >66.7% | 0% | 无法恢复（生病概率×1.5 > 1） |

**设计意图**：生病概率越高，恢复越困难。当健康值极低时（如20，生病概率90%），恢复概率为 1 - 90%×1.5 = 1 - 135% → 被截断为0%，即**无法自行恢复**，必须通过外部干预（清洁、使用道具等）提升健康值才能恢复。

## 6. 算法四：经验升级算法

### 6.1 经验值获取规则

| 操作 | 经验值 | 说明 |
|------|--------|------|
| 喂食 | +10 | 日常操作，经验适中 |
| 清洁 | +10 | 日常操作，经验适中 |
| 睡眠 | +5 | 被动操作，经验较少 |
| 玩耍 | +15 | 主动操作，经验最多 |
| 聊天 | +8 | 轻量操作，经验较少 |
| 使用道具 | 由道具定义 | 不同道具经验不同 |

### 6.2 升级所需经验公式

```
升级所需经验 = 当前等级 × 100 + 50
```

**各级别所需经验表**：

| 当前等级 | 升级所需经验 | 累计总经验 |
|----------|-------------|-----------|
| 1 → 2 | 150 | 150 |
| 2 → 3 | 250 | 400 |
| 3 → 4 | 350 | 750 |
| 5 → 6 | 550 | 2,250 |
| 10 → 11 | 1,050 | 7,750 |
| 20 → 21 | 2,050 | 27,750 |
| 50 → 51 | 5,050 | 162,750 |
| 99 → 100 | 9,950 | 512,550 |

### 6.3 升级检测算法

```
输入：VirtualPet pet
输出：更新 pet.level 和 pet.experience

步骤1：初始化
  if pet.level == null: pet.level = 1
  if pet.experience == null: pet.experience = 0

步骤2：循环升级检测
  currentLevel = pet.level
  currentExp = pet.experience
  expNeeded = currentLevel × 100 + 50

  while currentExp >= expNeeded 且 currentLevel < 100:
      currentExp -= expNeeded       // 扣除升级所需经验
      currentLevel++                // 等级+1
      pet.level = currentLevel      // 更新等级
      expNeeded = currentLevel × 100 + 50  // 计算下一级所需经验

  pet.experience = currentExp       // 更新剩余经验
```

**设计亮点**：使用 `while` 循环而非 `if` 判断，支持**连续多级跳升**。例如宠物在1级时通过使用高级道具获得了500经验，系统会自动从1级连续升到3级（150+250=400 < 500），剩余100经验保留。

## 7. 算法五：数值边界算法（clamp）

### 7.1 算法定义

```java
private long clamp(long value) {
    return Math.max(MIN_ATTR, Math.min(MAX_ATTR, value));
    // 等价于：Math.max(0, Math.min(100, value))
}
```

### 7.2 执行逻辑

```
if value < 0:
    return 0        // 下界截断
else if value > 100:
    return 100      // 上界截断
else:
    return value    // 原值返回
```

### 7.3 应用场景

clamp 函数在以下所有数值变更操作中被调用：

| 场景 | 调用位置 | 目的 |
|------|----------|------|
| 时间衰减后 | `applyTimeDecay` 中所有属性更新后 | 防止衰减导致负值 |
| 喂食后 | `feed` 中饥饿值和幸福感更新后 | 防止超过100 |
| 清洁后 | `clean` 中清洁度、健康值、幸福感更新后 | 防止超过100 |
| 睡眠后 | `sleep` 中能量和幸福感更新后 | 防止超过100 |
| 玩耍后 | `play` 中所有属性更新后 | 防止负值和超过100 |
| 聊天后 | `chatReward` 中幸福感和能量更新后 | 防止负值和超过100 |
| 使用道具后 | `useItem` 中所有属性更新后 | 防止负值和超过100 |

## 11. 算法执行顺序与依赖关系

### 11.1 时间衰减中的执行顺序

```
applyTimeDecay(pet)
  ├── 1. 前置校验（死亡/时间间隔）
  ├── 2. 计算经过时间 elapsedHours
  ├── 3. 计算饥饿值衰减 → 更新饥饿值 → clamp
  ├── 4. 计算能量衰减/恢复 → 更新能量 → clamp
  ├── 5. 计算清洁度衰减 → 更新清洁度 → clamp
  ├── 6. ★基于衰减后的新值★ 计算幸福感衰减 → 更新幸福感 → clamp
  ├── 7. ★基于衰减后的新值★ 计算健康值衰减/恢复 → 更新健康值 → clamp
  ├── 8. 记录变更日志（仅属性有变化时）
  ├── 9. recalculateStatus(pet) → 可能触发状态转换
  └── 10. 检查睡眠自动唤醒（能量 ≥ 100）
```

**关键依赖**：幸福感和健康值的衰减计算**依赖于**饥饿值、清洁度和能量衰减后的新值，因此计算顺序不可调换。

### 11.2 交互操作中的执行顺序

```
feed/clean/play/useItem(pet)
  ├── 1. 前置校验（死亡/睡眠/能量不足等）
  ├── 2. 保存旧值（用于变更记录和差异展示）
  ├── 3. 计算并更新属性值 → clamp
  ├── 4. 记录变更日志
  ├── 5. checkLevelUp(pet)（仅play/sleep/chat/useItem）
  ├── 6. recalculateStatus(pet)
  ├── 7. savePet(pet) → 持久化到数据库
  └── 8. 构建反馈消息和变更数据返回给前端
```

## 12. 设计亮点与注意事项

### 12.1 设计亮点

1. **惰性计算模式**：不在后台定时任务中逐秒计算衰减，而是在玩家访问时一次性计算累积衰减，大幅降低系统资源消耗
2. **连锁衰减机制**：低属性会加速其他属性的恶化，模拟真实宠物"忽视照护 → 恶性循环"的场景
3. **确定性伪随机**：生病/恢复判定使用基于宠物ID和时间窗口的伪随机数，确保公平且可重现
4. **概率模型分段设计**：健康值对生病概率的影响采用三段线性函数，在不同健康区间有不同的敏感度
5. **恢复门槛高于生病门槛**：生病容易恢复难，鼓励玩家主动照护而非被动等待
6. **连续升级支持**：while循环支持一次获得大量经验时连续升级多级

### 12.2 注意事项

1. **时间同步**：系统基于服务器时间计算衰减，确保服务器时间准确
2. **状态优先级**：死亡 > 睡眠 > 硬性生病 > 概率生病，判定链不可调换
3. **边界保护**：所有数值计算都经过clamp边界检查，防止异常值
4. **72小时衰减上限**：防止长时间未登录导致宠物属性全部归零
5. **1分钟最小间隔**：避免高频请求时的无效计算
6. **30分钟随机窗口**：防止玩家通过频繁刷新来"刷"状态变更
